## Two views of the same Python code

The audit script exists in **two forms**:

1. **`src/claude_free/`** — the canonical source. A real Python package with proper modules, types, and tests. Edit here.
2. **`claude-free-audit.py`** at repo root — a generated single-file bundle of the package. This is what gets shipped to user machines. Do **not** hand-edit; it has a "DO NOT EDIT" warning at the top.

`tools/build.py` produces (2) from (1):

```bash
python tools/build.py            # regenerate claude-free-audit.py
python tools/build.py --check    # exit 1 if it's stale (for CI)
```

The bundler:
- Concatenates modules in dependency order (`MODULE_ORDER` in `tools/build.py`).
- Strips internal `from claude_free.X import Y` lines (everything's in one namespace post-bundle).
- Strips per-module future-imports and module docstrings (one of each at the top).
- Collects external imports (stdlib only), de-dups them (merging `from X import a` and `from X import b` into one line), and emits them at the top.
- Wraps with shebang + header comment + `if __name__ == "__main__"` block.

After regenerating, `python -c "import ast; ast.parse(open('claude-free-audit.py').read())"` should succeed, and a smoke run (`--help`, `--version`) should work.

## Why a generated single-file artifact?

The deployed file has to satisfy two constraints:

1. **Zero runtime dependencies.** Users install with a curl one-liner. Asking them to `pip install` a package is friction.
2. **Single file fetchable from GitHub raw.** `claude-free update` is a single curl from `raw.githubusercontent.com/.../main/claude-free-audit.py`. A multi-file package would need a tarball or GitHub release flow.

Single-file bundling satisfies both. The cost is the build step and the rule that you don't edit the generated file. Worth it.

## Provider abstraction

`src/claude_free/providers/` is structured so adding a new inference backend (OpenRouter, Together AI, OpenAI, ...) is a focused change:

1. New module `src/claude_free/providers/openrouter.py` implementing `Provider` from `base.py`.
2. Register it in `providers/__init__.py`.
3. Add to `MODULE_ORDER` in `tools/build.py`.
4. Done — `--provider openrouter` becomes a valid CLI flag.

The `Provider` protocol requires:
- `name: str` — short CLI identifier.
- `api_base: str` — base URL.
- `proxy_value_prefix: str` — string prepended to model id when writing to `.env` (e.g. `"nvidia_nim/"` for NIM, `"openrouter/"` for OpenRouter).
- `list_models(timeout) -> list[str]` — fetch the catalogue.
- `probe(model, timeout, rate_limiter, retry_on_429) -> dict` — single streaming chat call, return TTFT + tokens.

The audit/calibrate logic in `commands/audit.py` only uses these methods. No NIM-specific code in there.

## How the launcher invokes the audit script

The bash launcher (Linux/macOS) `~/.local/bin/claude-free` does:

```bash
case "$1" in
  audit)     exec python3 ~/.local/bin/claude-free-audit.py "$@" ;;
  calibrate) exec python3 ~/.local/bin/claude-free-audit.py --set --early-exit --threshold 1000 "$@" ;;
  update)    curl -fsSL .../claude-free-audit.py -o ~/.local/bin/claude-free-audit.py ;;
  ...
esac
```

The Windows PowerShell launcher does the same with `& $py $AuditScript ...`.

So `claude-free calibrate` is just `claude-free-audit.py --set --early-exit --threshold 1000`. The deployed file's CLI also exposes a `calibrate` subcommand that sets the same defaults — both invocation styles work.

## Install-time wiring

The three installers (`install-claude-free.sh`, `install-claude-free-macos.sh`, `install-claude-free-windows.ps1`) each:

1. Fetch `claude-free-audit.py` from `raw.githubusercontent.com/.../main/claude-free-audit.py` to `~/.local/bin/claude-free-audit.py`.
2. Generate the launcher script (`~/.local/bin/claude-free` or `claude-free.ps1`) inline via heredoc / here-string.
3. The generated launcher has hardcoded paths to the audit script, the `.env` file, the proxy log, etc.

When the installer is re-run, it always rewrites both files — so installer changes propagate via re-install, audit-script changes propagate via `claude-free update`.

## Runtime data flow

```
                                                      ┌─────────────────────────┐
                                                      │ ~/.local/bin/claude-free│
                                                      │ (launcher)              │
                                                      └────────────┬────────────┘
                                                                   │ exec python3
                                                                   ▼
                                              ┌────────────────────────────────────┐
                                              │ ~/.local/bin/claude-free-audit.py  │
                                              │ (the bundled artifact)             │
                                              └────────────┬───────────────────────┘
                                                           │ reads NVIDIA_NIM_API_KEY
                                                           │ (env var or .env)
                                                           ▼
                       ┌────────────────────────────┐  ┌────────────────────────────┐
                       │ ~/free-claude-code/.env    │  │ https://integrate.api      │
                       │ (config: key + tier map)   │  │ .nvidia.com/v1/{models,    │
                       │                            │  │  chat/completions}         │
                       └─────────────▲──────────────┘  └────────────┬───────────────┘
                                     │ writes winner                │ probe results
                                     │ to MODEL_*                   │
                                     └──────────┐    ┌──────────────┘
                                                ▼    ▼
                                              ┌─────────────────┐
                                              │  audit.run_audit│
                                              │  ranks + picks  │
                                              └─────────────────┘
```

When the user actually runs Claude Code (`claude-free` with no args):

```
claude-free  ─────────► starts proxy on :8082 (uvicorn server:app)
                          │
                          │ reads .env tier mapping
                          │ MODEL_OPUS=nvidia_nim/<vendor>/<model>
                          │
                          ▼
                       proxy translates Anthropic API calls into NIM calls
                          ▲
                          │ http://localhost:8082
                          │
                          │ ANTHROPIC_BASE_URL=http://localhost:8082
                          │ ANTHROPIC_AUTH_TOKEN=freecc
                          │
claude-free ─exec─►    claude (the actual Claude Code CLI)
                          │
                          │ user's session
                          │
                          ▼
                       ~/.claude/  (sessions, history, projects)
```

So when you type a message in Claude Code:
1. `claude` sends an Anthropic-shaped request to `localhost:8082`.
2. The proxy reads the `model` field (`opus-4-7` etc.), maps it to `MODEL_OPUS` from `.env`.
3. The proxy converts the Anthropic request into an OpenAI-compatible request and forwards to NVIDIA NIM.
4. NIM responds; the proxy converts back to Anthropic format and streams to `claude`.
5. `claude` shows you the response with the label "Opus 4.7" — though it's actually `nvidia_nim/<whatever>` doing the work.

## Development workflow

```bash
git clone https://github.com/ChenghengLi/claude-free-installer
cd claude-free-installer

# Optional: install the package for local hacking + tests
python -m venv .venv && source .venv/bin/activate    # Windows: .venv\Scripts\activate
pip install -e ".[dev]"

# Edit src/claude_free/...

# Run tests
pytest

# Try the package directly (without bundling)
python -m claude_free audit --help

# Regenerate the deployed bundle (single Python file)
python tools/build.py

# Re-inject the launcher source into the 3 installers
python tools/build_installers.py

# Verify the bundle still works
python claude-free-audit.py --version

# Commit + push (re-running both build steps before each commit is the rule)
git add src/ tools/ tests/ claude-free-audit.py install-claude-free*.* docs/
git commit -m "..."
git push
```

The `--check` flag on both build scripts is for CI:

```bash
python tools/build.py --check              # exits 1 if claude-free-audit.py is stale
python tools/build_installers.py --check   # exits 1 if any installer is stale
```

## Adding a new subcommand

1. Add a module under `src/claude_free/commands/`. Export a `run_<name>` function.
2. Wire it into `cli.py`:
   - Import: `from claude_free.commands.<name> import run_<name>`.
   - Add a subparser in `build_parser()`.
   - Add a `if args.command == "<name>": return run_<name>(args, ...)` branch in `main()`.
3. Add the module to `MODULE_ORDER` in `tools/build.py`.
4. Add a `<name>)` case in all three launchers (installers).
5. Document in `docs/COMMANDS.md`.
6. Add a CHANGELOG entry.
7. `python tools/build.py && pytest && git commit && git push`.
